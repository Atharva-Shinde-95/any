Implementation · Stack · Storage · Solver
Technical Design
Framework choices, storage architecture, solver strategy, domain-agnostic design
SOLVER STRATEGY · SWAPPABLE DESIGN
Design principle: SolverOrchestratorAgent wraps the solver behind a standard interface. Swapping CP-SAT for another solver requires only changing the solver adapter — all agents, constraint formats, and KPI logic remain identical. The constraint registry outputs a solver-agnostic constraint object that each adapter translates to its native format.
SolverBest ForWhen to SwapStatus
CP-SAT (OR-Tools)Hard constraints, exact scheduling, integer problemsManufacturing, shift planning, job-shop● ACTIVE

STORAGE ARCHITECTURE
Redis HOT LAYER
· Active constraints (solver reads live)
· What-if scenario namespaces (isolated)
· LangGraph working state
· Evicted on scenario close / plan commit
· Write-through from MongoDB on plan trigger
MongoDB SOURCE OF TRUTH
· Full constraint registry (versioned)
· All rules: NL source + classified form
· Plan history + feedback log
· KPI definitions + results history
· Conflict resolution audit trail
Write path: User/Agent → MongoDB → sync to Redis  |  Read path: Solver → Redis only  |  What-If: Redis namespace clone (never touches MongoDB until committed)
TECH STACK
Orchestration
LangGraph
Graph-based agent transitions. Deterministic flow control. Human-in-loop nodes built-in. State shared across all agents via GlobalScheduleState.
Why not AutoGen
Our pipeline is structured, not open-ended negotiation. LangGraph gives explicit control over transitions — critical for scheduling correctness.
Rule Layer
Input modes
Plain English chat · Low-code form builder · Rule document upload (PDF/DOCX parsed)
LLM role
Claude/GPT-4 as RuleInterpreterAgent backbone — converts NL to structured constraint JSON. No hardcoded rules.
Solver Layer
OR-Tools
CP-SAT model receives constraint objects from registry. Variables = jobs/machines/time slots. Objective = user-defined KPI weights.
Adapter pattern
Each solver has an adapter class. Constraint registry outputs standard format. Adapter translates. Swap solver = swap adapter only.
Data Layer
MCP standard
Data contract defined once via MCP definition + UI field mapper. New data source = map fields, agent handles the rest.
Domain-agnostic
Entity map abstraction (Resource · Task · TimeSlot · Capacity · Constraint) works for manufacturing, healthcare, logistics, projects.
DOMAIN-AGNOSTIC ENTITY MODEL
Resource → Machine / Worker / Nurse / Vehicle / Equipment
Task → Job / Shift / Appointment / Delivery / Project Phase
TimeSlot → Shift / Window / Duration / Deadline / Calendar
Capacity → Load / Headcount / Throughput / Bandwidth
Constraint → Hard Rule / Soft Preference / Derived from Data
Domain changes: Update entity labels in UI mapper + domain-specific KPI templates. Agent logic, solver, and storage unchanged.

EXAMPLE · MANUFACTURING CONFLICT
CONFLICT DETECTED
Rule A (Hard): Machine M3 cannot run more than 8 hours without maintenance
Rule B (Hard): Job J-47 must complete in a single uninterrupted 10-hour run on M3
⚠ Both are Hard constraints on the same resource at the same time window. Cannot satisfy simultaneously.
OPTION 1
Relax Rule A → Soft
Allow M3 to run 10hrs with a maintenance flag raised. Schedule maintenance immediately after.
Impact: +15min delay post-job. Minor maintenance risk.
OPTION 2
Split Job J-47
Split into 8hr + 2hr segments with 30min maintenance break. Reschedule second segment.
Impact: +30min total. Job continuity broken. Check downstream.
OPTION 3
Reassign to M4
Move J-47 to Machine M4 which has no 8hr restriction. Check M4 availability window.
Impact: M4 capacity check needed. No maintenance risk.
CONFLICT TYPES THE SYSTEM HANDLES
Hard vs Hard
Two must-satisfy rules cannot both be true simultaneously. Agent must surface to human — cannot auto-resolve.
Hard vs Soft
Hard wins by default. Agent flags that soft preference was dropped and explains KPI impact. No human needed unless KPI breach is critical.
Soft vs Soft
Agent uses priority weights (1–10) set by user. Higher weight wins. AutoRankerAgent scores both options and suggests. Human can override.
Derived vs Any
Agent-inferred rule conflicts with user rule. Agent explains what it inferred and why. User either confirms derived rule or overrides it.
