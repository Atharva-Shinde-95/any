You are an expert AI architect and LangChain + LangGraph engineer.

Your task is to design and implement a session-based, multi-agent chatbot system
using LangChain, LangGraph, ChromaDB, and Streamlit.

The chatbot is designed ONLY for meeting transcript analysis (Teams, Zoom, Meet, etc.).
It must NOT behave as a general-purpose chatbot.

--------------------------------------------------
TECH STACK (MANDATORY)
--------------------------------------------------
- LangChain for agents, tools, and LLM interaction
- LangGraph for orchestration and state machine control
- ChromaDB for vector storage (session-scoped)
- Streamlit for UI (chat input + file upload)

--------------------------------------------------
HIGH-LEVEL ARCHITECTURE
--------------------------------------------------
The system MUST be a MULTI-AGENT SYSTEM, implemented using LangGraph,
with the following constraints:

- EXACTLY THREE AGENTS (LangChain agents)
- NO unnecessary graph nodes
- NO decision diamonds or flowchart logic
- Nodes represent conversation phases, not internal steps
- Internal logic (classification, duplicate detection, validation)
  MUST live inside agents using tools

--------------------------------------------------
AGENTS (NODES IN LANGGRAPH)
--------------------------------------------------

AGENT 1: InputAgent  (ENTRY POINT)
---------------------------------
Purpose:
- Acts as the controller and intent-understanding agent
- Runs on EVERY user input

Responsibilities:
- Classify user intent using an LLM into:
  - greeting
  - invalid_input
  - transcript_input
  - follow_up_question
- Politely respond to greetings
- Politely reject invalid / out-of-scope inputs
- Decide routing to:
  - IngestionAgent (for transcript input)
  - ChatAgent (for follow-up questions)
- Loop back to itself when user input is incomplete or invalid
- Call Session Manager Tool when session creation/loading is needed

Rules:
- Greeting and invalid input MUST NOT end the graph
- The agent must always ask the user for correct input if input is invalid
- This agent must NOT store embeddings or access ChromaDB directly

--------------------------------------------------

AGENT 2: IngestionAgent
-----------------------
Purpose:
- Process a meeting transcript exactly once per session

Responsibilities:
- Handle both:
  - pasted plain text transcript
  - uploaded files (PDF, DOCX, TXT)
- Extract text if file-based
- Clean and normalize transcript text
- Validate that the content is a meeting transcript
- Perform DUPLICATE TRANSCRIPT DETECTION:
    - Generate a content-based hash of the transcript
    - Use Session Manager Tool to check if transcript already exists
- Behavior on duplicate:
    - DO NOT re-index
    - Load existing session
    - Route directly to ChatAgent
- Behavior on new transcript:
    - Chunk text
    - Generate embeddings
    - Store embeddings into ChromaDB under session namespace
- If transcript is invalid or rejected:
    - Explain politely
    - Route user back to InputAgent

IMPORTANT:
- Duplicate detection MUST NOT be a separate LangGraph node
- It must be internal logic inside IngestionAgent
- This agent represents the "ingestion phase" only

--------------------------------------------------

AGENT 3: ChatAgent
------------------
Purpose:
- Answer all follow-up questions based on uploaded transcript

Responsibilities:
- Perform Retrieval-Augmented Generation (RAG)
- Retrieve context ONLY from the ChromaDB collection
  associated with the active session
- Answer questions strictly grounded in transcript content
- If answer does not exist in transcript, say so clearly
- After each response:
    - Ask user whether to continue, start a new session, or end session
- Route:
    - Continue → InputAgent
    - End session → END

Rules:
- This agent must NOT answer general knowledge questions
- This agent must NOT hallucinate beyond retrieved context

--------------------------------------------------
LANGGRAPH STRUCTURE (MANDATORY)
--------------------------------------------------

The LangGraph MUST contain exactly the following nodes:

- InputAgent  (entry point)
- IngestionAgent
- ChatAgent

The control flow MUST loop until the user explicitly ends the session.

Valid LangGraph mental model:

[*] → InputAgent
InputAgent → InputAgent
InputAgent → IngestionAgent
InputAgent → ChatAgent
IngestionAgent → InputAgent
IngestionAgent → ChatAgent
ChatAgent → InputAgent
ChatAgent → END

NO extra nodes for:
- greeting handling
- invalid input handling
- duplicate detection
- session checking

--------------------------------------------------
TOOLS (NOT AGENTS)
--------------------------------------------------

The following must be implemented as TOOLS, not agents:

1. Session Manager Tool
   - Create session
   - Load session
   - Store transcript hash
   - Detect duplicate transcript
   - Return session status

2. Document Loader Tools
   - PDF loader
   - DOCX loader
   - TXT loader

3. Text Cleaning Utility
   - Normalize whitespace
   - Remove noise

4. Embedding Tool
   - Generate embeddings

5. ChromaDB Tool
   - Store embeddings by session namespace
   - Retrieve embeddings by session namespace

--------------------------------------------------
SESSION & DUPLICATE RULES
--------------------------------------------------
- Each session has exactly one transcript
- Transcript uniqueness is determined by content hash
- Same transcript uploaded again → same session reused
- Each session maps to one ChromaDB collection/namespace
- Sessions are user-scoped and isolated

--------------------------------------------------
STREAMLIT UI REQUIREMENTS
--------------------------------------------------
- Chat interface using st.chat_input
- File upload using st.file_uploader
- Display assistant responses
- Do NOT implement logic in Streamlit
- Streamlit only forwards inputs to LangGraph

--------------------------------------------------
IMPORTANT DESIGN PRINCIPLES
--------------------------------------------------
- Agents THINK, tools EXECUTE
- Nodes represent phases, not steps
- Never over-engineer LangGraph
- Always loop unless user explicitly ends session
- System must remain bounded to meeting transcript analysis

--------------------------------------------------
DELIVERABLE EXPECTATION
--------------------------------------------------
Design the system following all above constraints.
The final implementation must be:
- Clean
- Minimal
- Fully agentic
- Correctly orchestrated
- Production-ready in structure
